네. 지금 상황이면 **“웹으로 만든 데모를 버리고 처음부터 네이티브 앱처럼 가자”**가 아니라,
**“이미 만든 웹 데모를 ‘모바일 게임 QA 포트폴리오용 산출물’로 재포지셔닝해서 모바일 감각을 강하게 입히는 방향”**이 제일 좋습니다.

우선 제가 본 당신 repo는 이겁니다:

* 포트폴리오 repo: [bps-demo](https://github.com/jaehyun4400/bps-demo)
* 현재 README 기준으로는 **“동양 궁중 판타지 모바일 수집형 방치 RPG를 브라우저 게임 데모로 구현”**했고, QA 산출물(TEST PLAN / TEST CASES / BUG REPORT / TRACEABILITY / TEST REPORT)까지 붙어 있습니다. 즉, **“게임 QA 포폴의 뼈대”는 이미 상당히 잘 잡혀 있습니다.** ([GitHub][1])

그래서 결론부터 말하면:

---

## 결론: 당신에게 가장 좋은 시나리오

### **“모바일 게임 QA 포트폴리오용 플레이어블 프로토타입”으로 포지셔닝을 바꾸십시오.**

즉, **APK/IPA까지는 안 가되, 보는 사람이 ‘아, 이 사람은 모바일 게임 QA 관점으로 게임을 설계하고 검증했구나’라고 느끼게 만드는 것**이 핵심입니다.

웹이냐 앱이냐 자체가 본질이 아닙니다.
**게임회사 QA 포트폴리오에서 더 중요한 건 아래 4개입니다.**

1. **이 사람이 모바일 게임의 핵심 품질 포인트를 아는가**

   * 튜토리얼 흐름
   * 재화/성장 밸런스
   * 소환/확률/픽업 UX
   * 전투/보상 루프
   * 결제/상점/이벤트/출석/우편/푸시성 UX
   * 발열/배터리/프레임/로딩/복귀/네트워크 이슈 등

2. **이 사람이 QA 문서를 “게임 서비스 기준”으로 쓸 수 있는가**

   * 기능 테스트케이스만 있는지
   * 아니면 **라이브 운영 관점 / 업데이트 회귀 / 밸런스 리스크 / CS 리스크**까지 보는지

3. **이 사람이 버그를 찾는 사람을 넘어서, 품질 전략을 세울 수 있는가**

   * “무엇을 먼저 검증해야 하는가”
   * “어떤 결함이 런칭 리스크인가”
   * “업데이트 후 어디서 회귀가 날 확률이 높은가”

4. **이 사람이 실제로 손으로 만져본 결과물을 보여주는가**

   * 말로만 “모바일 게임 QA 하고 싶어요”가 아니라
   * **플레이 가능한 프로토타입 + QA 산출물 + 결함 분석 + 개선 제안**이 한 세트로 있는가

당신은 이미 4번의 절반 이상을 해놨습니다.
문제는 **“이게 웹게임처럼 보인다”**는 포장 방식이지, 본질이 아닙니다.

# 시나리오 A — 가장 추천

# **“모바일 게임처럼 보이는 웹 기반 플레이어블 QA 포트폴리오”**

이게 제 1추천입니다.
이유는 **개발 부담 대비 포트폴리오 효율이 가장 좋기 때문**입니다.

## 핵심 컨셉

현재 HTML 데모를 버리지 말고, 아예 이렇게 명명하십시오.

> **모바일 수집형 RPG QA 포트폴리오용 플레이어블 프로토타입**
> *Web-based, Mobile-first Interactive Demo*

즉, “웹게임”이 아니라 **“모바일 게임 QA 검증을 위한 시뮬레이션 데모”**로 정의를 바꾸는 겁니다.

## 이 시나리오에서 반드시 바꿔야 할 것

## 1) 화면 비율과 UX를 **모바일 퍼스트**로 고정

지금 가장 먼저 손봐야 할 건 기술이 아니라 **보이는 방식**입니다.

### 해야 할 것

* 기본 해상도를 **세로형 모바일 기준**으로 고정

  * 예: 390x844, 412x915 같은 모바일 뷰 기준
* 데스크탑에서 열어도

  * 중앙에 **휴대폰 프레임 안에서 게임이 돌아가는 것처럼** 보이게
* 클릭보다는 **탭 중심 UX**로 문구를 통일

  * “클릭” → “탭”
* hover 의존 제거
* 버튼 크기 / 터치 영역 / 스크롤 / 팝업 / 뒤로가기 흐름을 모바일처럼 정리

### 왜 중요하냐

면접관이 링크를 열었을 때
“웹페이지네?”가 먼저 나오면 손해입니다.

반대로 첫 화면에서

* 세로형 모바일 화면
* 로비 UI
* 하단 메뉴바
* 배너 / 우편 / 상점 / 소환 / 던전 구조

이렇게 보이면 **“모바일 게임 포폴이네”**로 인식됩니다.

즉, **실체보다 첫인상이 중요합니다.**

---

## 2) README와 포폴 소개 문구에서 “웹 구현”을 앞세우지 말 것

지금 README에 “브라우저 게임 데모”라는 표현이 있는데, 이건 약간 손해입니다.
사실은 틀린 말은 아닌데, **당신이 팔아야 할 포인트는 ‘웹 구현력’이 아니라 ‘모바일 게임 QA 관점’**이거든요.

### 표현을 이렇게 바꾸는 게 좋습니다

기존 느낌:

* “브라우저 게임 데모”

추천 느낌:

* **모바일 수집형 RPG QA 포트폴리오용 플레이어블 프로토타입**
* **모바일 게임 서비스 QA를 가정해 설계한 인터랙티브 데모**
* **가상의 모바일 라이브 게임 업데이트(v1.2 → v1.3)를 대상으로 QA 문서와 검증 시나리오를 구성한 프로젝트**
* **실제 모바일 게임 QA 관점의 테스트 범위를 설명하기 위해 제작한 실행형 데모**

즉,
**“웹으로 만들었다”는 건 구현 수단**이고,
**“모바일 게임 QA 포트폴리오를 위해 만들었다”는 게 정체성**이어야 합니다.

## 3) QA 산출물을 “게임회사 QA가 보고 싶어할 구조”로 재편

당신 repo에는 이미 QA 문서가 있습니다. 이건 큰 장점입니다. ([GitHub][1])
그런데 게임회사 이직용이라면 **문서의 결을 게임 서비스형으로 더 밀어야 합니다.**

### 포폴에서 보여줘야 할 문서 묶음

1. **게임 소개서**

   * 장르 / 핵심 루프 / BM 구조 / 재화 구조 / 성장 구조 / 콘텐츠 구조
   * 타깃 유저 / 1일 플레이 루프 / 세션 길이 / 과금 포인트

2. **QA 전략서**

   * 무엇을 우선 검증할지
   * 왜 그걸 먼저 검증하는지
   * 어떤 영역이 런칭 리스크인지
   * 어떤 영역이 라이브 업데이트 회귀 포인트인지

3. **테스트케이스**

   * 기능 테스트
   * 예외 테스트
   * 경계값 테스트
   * 재현 절차 명확한 버그 재현 시나리오

4. **버그 리포트**

   * 증상 / 재현 절차 / 기대결과 / 실제결과 / 심각도 / 우선순위 / 영향 범위
   * 스크린샷/GIF
   * “왜 이 버그가 치명적인가”까지

5. **업데이트 검증 문서**

   * v1.2 → v1.3 변경점 기반 회귀 범위
   * 영향도 분석
   * smoke / sanity / full regression 구분

6. **라이브 운영 QA 관점 문서**

   * 출석 이벤트
   * 우편 보상
   * 픽업 배너 종료/갱신
   * 리딤코드 만료
   * 점검 후 보상 지급
   * 시간대/일일 초기화/주간 초기화

# 시나리오 B — 괜찮지만 A보다 덜 추천

# **“모바일 게임 기획서 + Figma 프로토타입 + QA 문서” 조합**

이건 개발 부담이 낮습니다.
대신 **“실제로 플레이 가능하다”는 강점이 약해집니다.**

## 구성

* Figma로 모바일 화면 흐름 제작
* 로비 → 편성 → 소환 → 던전 → 결과 → 우편 → 상점 흐름 연결
* 각 화면별 QA 포인트 명시
* 여기에 테스트케이스/버그리포트/릴리즈 검증 문서 붙이기

## 장점

* 구현 부담 적음
* 화면 설계와 UX 설명이 쉬움
* 모바일 느낌 내기 쉬움

## 단점

* 당신은 이미 **플레이 가능한 데모**를 만들어본 사람입니다.
* 그런데 여기서 Figma-only로 내려오면 오히려 **강점을 버리는 꼴**이 됩니다.

그래서 이건
**“도저히 데모를 더 손볼 시간이 없다”**면 차선책이지, 최선은 아닙니다.

# 시나리오 C — 가능은 하지만 비추천

# **Unity나 Flutter로 아주 얕은 모바일 앱 데모를 새로 만드는 것**

겉보기엔 제일 “모바일스러워” 보이죠.
그런데 저는 지금 이 선택은 비추천합니다.

## 왜 비추천하냐

당신의 목적은 **게임 클라이언트 개발자 포트폴리오**가 아니라 **게임 QA 이직용 포트폴리오**입니다.

새로 Unity/Flutter 잡으면 생기는 일:

* 엔진/프레임워크 적응 시간 듦
* 화면 구현 시간 듦
* 리소스/빌드/배포 관리 생김
* 결국 QA 포폴이 아니라 **개발 삽질 포폴**이 되기 쉬움

그리고 더 중요한 문제:

* 면접관이 보고 싶은 건 “APK 만들 줄 아는 QA”가 아니라
* **“게임 품질을 어떤 시야로 검증하는 QA냐”**입니다.

즉, **노력 대비 효율이 너무 떨어집니다.**

# 2층: “QA가 이 게임을 어떻게 검증했는가”

여기가 진짜 본체입니다.

## 섹션 1. QA 전략

이런 식으로 써야 합니다.

### 예시 목차

* 왜 이 게임에서 **전투/보상/소환/재화**를 최우선 검증 대상으로 잡았는가

* 어떤 결함이 유저 이탈과 직결되는가
* 어떤 영역이 **CS 폭탄**으로 이어질 수 있는가

### 예시 문장

* “수집형 RPG에서 재화 지급 오류와 소환 결과 불일치는 과금 신뢰도에 직접적인 영향을 주므로 최상위 검증 대상으로 분류하였다.”
* “방치형 구조상 오프라인 보상/일일 초기화/출석 보상은 재현 빈도가 높고 사용자 민감도가 높아 회귀 테스트 우선순위를 상향하였다.”

이런 문장이 있어야 **10년차 QA답습니다.**

* 업적 달성 시점 불일치
* 보상 수령 후 UI 갱신 실패

## 6. 라이브 운영/이벤트 테스트

* 이벤트 시작/종료 시각
* 점검 보상 우편 발송
* 배너 교체
* 핫타임 적용 시간
* 리딤코드 만료/중복 사용
* 타임존 이슈

## 7. 환경/단말 관점 테스트

APK를 안 만들더라도 **포폴 문서에는 반드시 넣어야 합니다.**

* 저사양 단말 가정
* 화면비 이슈
* 터치 오입력
* 발열/배터리/메모리 사용 가정
* 백그라운드 → 포그라운드 복귀
* 푸시/딥링크가 있다면 진입 흐름

## 8. 회귀 테스트 전략

* v1.2 → v1.3에서 바뀐 기능 목록
* 영향받는 시스템
* 재검증 범위
* Smoke / Full Regression 구분
* 출시 보류 기준

# 제가 특히 추천하는 “모바일 게임답게 보이게 하는 장치”

이건 꽤 중요합니다.

## 장치 1) **가짜 앱 빌드 정보/패치 정보 UI**

예:

* App Version 1.3.0
* Build 2026.06.24
* Patch Note 버튼
* 문의/고객센터 버튼
* 약관/계정/로그아웃/서버 선택 UI

이런 요소가 있으면 **웹 페이지**가 아니라 **운영 중인 모바일 게임 클라이언트**처럼 보입니다.

---

## 장치 2) **“모바일 게임 QA 체크포인트”를 화면 옆에 붙이지 말고 문서로 분리**

웹 화면 옆에 설명이 너무 많으면 포폴이 아니라 튜토리얼 페이지처럼 보입니다.
게임 화면은 게임 화면답게 두고, QA 설명은 아래 문서로 빼는 게 낫습니다.

### 예

* Demo
* Game Spec
* Test Strategy
* Test Cases
* Bug Reports
* Release Validation
* Postmortem

이렇게 탭 분리.

---

## 장치 3) **실제 결함 사례를 5개 정도 “대표 버그 카드”로 보여주기**

예:

* BUG-001: 특정 조건에서 소환 연출 종료 후 결과 팝업 미노출
* BUG-002: 우편 일괄 수령 시 재화 반영 지연
* BUG-003: 일일 초기화 직후 출석 보상 중복 수령 가능
* BUG-004: 전투 패배 후 재도전 시 버프 상태 초기화 누락
* BUG-005: 이벤트 배너 종료 후 잘못된 진입 링크 유지

여기서 중요한 건 단순 버그 나열이 아니라:

* **유저 영향**
* **비즈니스 영향**
* **재현 조건**
* **우선순위 이유**

까지 적는 겁니다.

# 여기서 아주 중요한 포인트 하나

## 포트폴리오에 **“못한 것”도 QA답게 써야 합니다.**

예를 들면 이런 식입니다.

### 나쁜 표현

* “APK는 못 만들었습니다.”
* “실기기 테스트는 못 했습니다.”

### 좋은 표현

* “본 프로젝트는 QA 포트폴리오용 플레이어블 프로토타입으로, 실제 배포 빌드 대신 웹 기반 모바일 시뮬레이션 형태로 제작하였다.”
* “실기기 빌드가 없는 한계로 인해 발열/메모리/OS 권한/스토어 결제/푸시 알림 검증은 범위에서 제외하였다.”
* “대신 모바일 라이브 게임 QA에서 중요도가 높은 튜토리얼, 재화, 소환, 보상, 이벤트, 업데이트 회귀 시나리오를 중심으로 검증 범위를 설계하였다.”

이건 핑계가 아니라 **테스트 범위 정의**입니다.
QA스럽고, 훨씬 전문적으로 보입니다.
