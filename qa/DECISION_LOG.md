# 의사결정 로그 (Decision Log)

프로젝트를 진행하며 내린 주요 판단과, 그때 함께 검토했던 다른 방안을 기록한다.

"무엇을 만들었는가"보다 "왜 그렇게 결정했는가"를 남기는 것이 목적이다.

---

## 1. QA 문서를 누적형에서 버전별 폴더 분리로 전환

[배경]

처음에는 TEST_CASES / TRACEABILITY를 버전 구분 없이 한 파일에 계속 쌓는 누적형(하이브리드) 구조로 진행. v1.3 항목까지 더하니 TC 번호가 51개까지 늘었고, "이 TC가 어느 업데이트 대상인지"가 한눈에 안 들어오는 상황이 발생.

[결정]

`qa/1.2/`, `qa/1.3/` 폴더로 완전히 분리하고, 폴더마다 TC/Bug 번호를 1번부터 다시 시작.

[이유]

릴리즈 단위로 보고서를 묶는 게 실제 운영 QA의 작업 단위에 더 가까움. 면접관이 "v1.3에서 뭘 검증했나"를 물으면 폴더 하나만 열면 되도록 만들고 싶었음.

[다른 후보안]

누적 유지 + 태그로 버전 표시 → 번호 체계가 한 번 꼬이면 추적이 더 어려워져 폐기.

[절충안]

같은 BUG-001이 1.2와 1.3 양쪽에 존재하게 됨. 그래서 항상 폴더 경로와 함께 언급하기로 규칙을 정함.

---

## 2. 의도적 결함을 '재화·확률 정합성' 영역에 심은 이유

[배경]

데모에 의도적 버그를 심어 QA가 그것을 잡아내는 과정을 보여주기로 함. 버그를 아무 데나 심을 수도 있었지만, 어디에 심느냐가 곧 "무엇을 중요하게 보는가"의 표현이라고 판단.

[결정]

의도적 버그를 모두 재화/확률 정합성 영역으로 설정. v1.2/v1.3에 걸쳐 일관되게 적용함.
- v1.2 BUG-001 (Critical): 이벤트 재화 교환 연속 터치 시 보상 중복 지급 — 재화 부정 획득(과다 지급)
- v1.3 BUG-001 (Major): 출석 7일 사이클이 리셋되지 않아 보상 영구 차단 — 받아야 할 재화 미지급
- v1.3 BUG-002 (Major): 10+1 소환이 11회가 아닌 10회만 지급(보너스 누락)

[이유]

수집형 RPG에서 재화 지급 오류와 소환 결과 불일치는 과금 신뢰도에 직접 타격을 줌.

재화 정합성은 양방향 리스크 — 유저가 부당하게 더 받는 것(부정 획득)과 받아야 할 걸 못 받는 것(미지급/차단)을 모두 심어 검증 범위를 넓힘.

기능이 "안 되는" 버그보다, 유저가 손해를 보고도 모르거나 운영 CS로 번지는 버그가 더 위험하다고 판단.

[다른 후보안]

UI 깨짐 같은 가시적 버그 → 눈에 잘 띄어 잡기 쉽지만, "왜 치명적인가"를 설명할 여지가 적어 제외.

## 3. 시즌 전환 시, 과거 문서를 고치지 않고 회귀 섹션을 새로 만든 결정

[배경]

v1.3에서 발렌타인 이벤트 자동 팝업을 제거함. 그 결과 v1.2 테스트케이스 중 "이벤트 팝업이 1회 노출된다"(TC-011/012)가 현 스펙과 맞지 않게 됨.

[결정]

v1.2 문서는 그대로 두고(과거 시점의 기록으로 보존), v1.3 문서에 "시즌 전환 회귀" 섹션을 새로 만들어 "이제는 팝업이 노출되지 않아야 한다"를 새 TC로 검증함.

[이유]

과거 보고서를 소급해서 고치면 "그때 무엇을 검증했는지"의 기록이 사라짐. QA 문서는 그 시점의 사실을 남기는 것이지, 항상 최신 상태로 덮어쓰는 게 아니라고 판단.

## 4. 게임 세이브 데이터 마이그레이션 — 버전 prefix를 바꾸되 기존 키 함께 조회

[배경]

데모는 플레이어의 게임 진행 상태(닉네임·진행도·보유 재화 등)를 브라우저 localStorage에 저장하고, 저장 키 앞에 버전 prefix를 붙여 구분함. 그런데 빌드는 v1.3.0인데 이 prefix가 `bps_v120_`로 남아 버전과 안 맞았음. `bps_v130_`로 바꾸고 싶었지만, prefix만 그냥 교체하면 v1.2 때 저장된 기존 키를 못 찾아 그동안의 계정 데이터(세이브)가 통째로 끊김.

[결정]

신규 키는 `bps_v130_`로 쓰되, 로드/닉네임 조회는 신규+기존(`bps_v120_`) 양쪽을 모두 보도록 함. 삭제 시에도 양쪽 키를 함께 지움.

[이유]

라이브 게임에서 업데이트로 인해 기존 유저 데이터가 깨지는 것은 있어서는 안 되는 사고임. 데모지만 같은 원칙을 적용해, "기존 데이터 마이그레이션"을 기본값으로 두고 싶었음.

[검증]

기존(구버전) 키로 저장된 상태에서 로드 → 재저장 시 신규 키가 생성되는지까지 확인함.

## 5. 포트폴리오의 정체성을 '웹 데모'에서 'QA 프로젝트'로 재포지셔닝

[배경]

README가 "브라우저 게임 데모"를 앞세우고 있었음. 이렇게 두면 평가 축이 '웹 구현력'으로 잡힘.

[결정]

첫 문장을 "모바일 게임 QA 포트폴리오용 플레이어블 프로토타입"으로 바꾸고, 웹/단일 파일이라는 사실은 "검증 환경"으로 한 단계 내림.

[이유]

이 프로젝트로 증명하려는 건 QA 관점이지 프론트엔드 구현이 아님. 도구(웹)가 정체성을 가리면 안 된다고 판단.

## 6. 성능 측정치를 '가정값'으로 명시한 결정

[배경]

v1.2 TEST_REPORT에 기기별 FPS/메모리/발열 측정표가 있음. 그런데 이 프로젝트는 실제 빌드가 없는 웹 프로토타입이라, 그 수치는 실측이 아님.

[결정]

표를 지우지 않고, 그 위에 "실측이 아닌 라이브 환경 가정 시뮬레이션 값"이라는 단서를 명시함. 실서비스라면 어떻게 실측했을지는 회고 문서에 따로 정리함.

[이유]

검증 안 한 수치를 검증한 것처럼 보이게 두는 건 QA로서 신뢰를 깎는 일임. "무엇을 가정했는지"를 드러내는 게 오히려 범위를 정확히 안다는 증거라고 판단.
