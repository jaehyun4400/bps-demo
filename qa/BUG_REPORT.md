## BUG-001

### 기본 정보

| 항목 | 내용 |
|---|---|
| Bug ID | BUG-001 |
| 제목 | 이벤트 재화 교환 연속 터치 시 보상 중복 지급 |
| 심각도 | Critical |
| 우선순위 | P1 |
| 발견일 | 2026-04-22 |
| 발견자 | 박재현 |
| 상태 | Open |
| 연결 TC | TC-013 (FAIL) |

### 환경

| 항목 | 내용 |
|---|---|
| 기기 | 갤럭시 S24 (Android 14) |
| 빌드 | v1.2.0-rc1 |
| 서버 | Staging |

### 재현 절차

1. 로비 → 이벤트 탭 진입
2. '두근두근 발렌타인' 이벤트 페이지 진입
3. 초콜릿 상자 보유 상태에서 재화 교환 버튼을 빠르게 연속으로 탭 (0.2초 이내 3회 이상)
4. 지급된 재화 수량 확인

### 기대 결과

보상이 1회만 지급되며, 연속 터치에 대한 중복 처리 방지

### 실제 결과

터치 횟수만큼 보상이 중복 지급됨 (3회 탭 시 3배 보상)

### 원인 분석

서버에 교환 요청을 전송한 후 응답을 받기 전에 버튼이 재활성화되어 중복 요청이 발생함. 클라이언트 측 버튼 Disable 처리가 누락된 상태로, 서버의 중복 방지 로직도 부재한 것으로 추정됨.

### 영향 범위

이벤트 재화 교환 화면 전체. 단기간에 과도한 재화 지급으로 게임 경제 밸런스 붕괴 가능.

### 조치 방향

- 교환 버튼 클릭 시 즉시 비활성화(Disable) 처리
- 서버 응답 완료 후 버튼 재활성화
- 서버 측 동일 요청 중복 처리 방지 로직 추가 검토

### 첨부

> ℹ️ BUG-001은 서버 응답 전 중복 요청이 핵심 원인으로, 브라우저 단독 데모에서는
> 재현이 불가합니다. 실제 테스트는 Staging 서버 연동 환경(갤럭시 S24)에서 수행되었으며
> 첨부 영상 및 스크린샷은 해당 기기에서 수집된 파일로 이 저장소에 포함되지 않습니다.

## BUG-002

### 기본 정보

| 항목 | 내용 |
|---|---|
| Bug ID | BUG-002 |
| 제목 | 대례복 장착 후 상태창 전투력 UI 미갱신 |
| 심각도 | Major |
| 우선순위 | P1 |
| 발견일 | 2026-04-22 |
| 발견자 | 박재현 |
| 상태 | Open |
| 연결 TC | TC-008 (FAIL) |

### 환경

| 항목 | 내용 |
|---|---|
| 기기 | 갤럭시 S24 (Android 14), 아이폰 14 Pro (iOS 17) |
| 빌드 | v1.2.0-rc1 |
| 서버 | Staging |

### 재현 절차

1. 편성 탭 진입 — 헤더의 ⚡ 전투력 수치 확인 (예: **458**)
2. 의복 탭 클릭
3. 대례복 카드 → '착용하기' 클릭
4. 헤더의 ⚡ 전투력 수치 확인 — **변화 없음 (버그)** (여전히 458)
5. 로비로 이동 후 편성 탭 재진입
6. 헤더의 ⚡ 전투력 수치 확인 — **538로 갱신됨 (정상)**

### 기대 결과

대례복 장착 즉시 전투력 수치가 갱신됨

### 실제 결과

- 장착 직후: 전투력 수치가 이전 값으로 그대로 표시됨
- 상태창 재진입 후: 갱신된 전투력 수치 정상 표시

### 원인 분석

실제 능력치 계산(`calcCP()`)은 장착 즉시 정상 적용되고 있으나, 의복 장착 이벤트 핸들러에서 대례복(`daerye`) 한정으로 전투력 UI 갱신 함수(`updateCP()`) 호출이 누락됨. 화면 재진입 시 UI가 재초기화되며 정상 표시되는 점에서 렌더 트리거 누락으로 판단.

### 영향 범위

의복 장착 후 편성 헤더의 ⚡ 전투력 수치 표시에만 영향. 실제 전투 데미지 계산에는 영향 없음. 단, 유저가 표시된 전투력을 기준으로 의사결정하는 만큼 신뢰도에 영향을 줌.

### 조치 방향

- 의복 장착 완료 이벤트 핸들러에 `updateCP()` 호출 추가 (대례복 분기 포함)
- 장착 / 해제 양방향 모두 갱신 여부 확인

### 재현 가능 여부

데모에서 직접 재현 가능 — 편성 탭 의복 탭에서 대례복 장착 후 헤더의 전투력 수치 즉시 확인

| `attachments/BUG-002_2_after_equip.png`  | 대례복 착용 직후 — '착용중' 표시이나 전투력 여전히 458 (버그) |

## BUG-003

### 기본 정보

| 항목 | 내용 |
|---|---|
| Bug ID | BUG-003 |
| 제목 | 출석 이벤트 Day 7 수령 후 사이클 리셋 불가 (출석 버튼 영구 비활성화) |
| 심각도 | Major |
| 우선순위 | P1 |
| 발견일 | 2026-06-19 |
| 발견자 | 박재현 |
| 상태 | Open |
| 연결 TC | TC-016 (FAIL) |

### 환경

| 항목 | 내용 |
|---|---|
| 기기 | 갤럭시 S24 (Android 14) / 브라우저 데모 |
| 빌드 | v1.3.0 (여름방학맞이 7days 특별 출석이벤트) |
| 서버 | Staging |

### 재현 절차

1. 출석 이벤트 팝업에서 Day 1 ~ Day 7 보상을 하루 1회씩 7일에 걸쳐 모두 수령
   - (데모 빠른 재현: Day 6 수령 완료 상태에서 Day 7 수령)
2. "이번 사이클: 7/7일 완료" 표시 및 전 슬롯 ✔ 처리 확인
3. **다음 날(자정 경과 후) 출석 팝업 재진입**
4. 출석 수령 버튼 상태 확인

### 기대 결과

새 사이클로 리셋되어 Day 1 보상부터 다시 수령 가능 (버튼 활성, "이번 사이클: 0/7일")

### 실제 결과

- "이번 사이클: 7/7일 완료" 상태로 고착
- 새 날짜가 되어 수령 가능 조건(`canCheckInToday()`)은 충족되지만, 수령 버튼이 **"수령 완료" 상태로 비활성화**되어 다음 사이클 진입 불가
- 이후 매일 진입해도 동일 — 출석 보상을 **영구적으로 받을 수 없음**

### 원인 분석

`claimCheckIn()`에서 Day 7 수령 후 `checkInDay`를 0으로 되돌리는 사이클 리셋 코드가 누락됨.

```javascript
S.checkInDay++;                                // Day7 수령 → checkInDay = 7
// if (S.checkInDay >= CHECKIN_REWARDS.length) S.checkInDay = 0;  // ← 이 리셋 라인 누락
```

수령 버튼 활성 조건이 `canCheckInToday() && S.checkInDay < CHECKIN_REWARDS.length(=7)` 인데,
`checkInDay`가 7에서 리셋되지 않아 `checkInDay < 7` 조건이 영구히 `false` → 버튼 영구 비활성화.

### 영향 범위

출석 이벤트 기능 전체. 7일 사이클을 1회 완주한 모든 유저가 이후 출석 보상을 전혀 받지 못함. 신규 유저보다 **장기 접속 유저(충성 유저)에게 직접 피해**가 발생하여 이탈 위험이 큼.

### 조치 방향

- `claimCheckIn()` 내 Day 7 수령 직후 `checkInDay >= 7` 이면 `checkInDay = 0`으로 리셋
- 리셋 후 첫 진입 시 Day 1 보상 정상 수령되는지 회귀 확인
- 경계값 TC(Day 7 → Day 1 전환) 자동화 검토

### 재현 가능 여부

데모에서 직접 재현 가능 — 출석 팝업 Day 7까지 수령 후, 날짜 변경 시점에 버튼 비활성 확인

| `attachments/CHECKIN_2_claimed.png`      | Day 1 수령 직후 — 골드 510,000, 슬롯 dim+✔, 당일 중복 방지 |

## BUG-004

### 기본 정보

| 항목 | 내용 |
|---|---|
| Bug ID | BUG-004 |
| 제목 | 혜 귀인 픽업 배너 10+1 소환 시 11회가 아닌 10회만 지급 (보너스 1회 누락) |
| 심각도 | Major |
| 우선순위 | P1 |
| 발견일 | 2026-06-19 |
| 발견자 | 박재현 |
| 상태 | Open |
| 연결 TC | TC-017 (FAIL) |

### 환경

| 항목 | 내용 |
|---|---|
| 기기 | 갤럭시 S24 (Android 14) / 브라우저 데모 |
| 빌드 | v1.3.0 (혜 귀인 픽업 배너) |
| 서버 | Staging |

### 재현 절차

1. 로비 → 🌸 픽업소환 진입
2. 다이아 1,800 이상 보유 상태에서 **"10+1 소환"** 버튼 클릭
3. 소환 결과 팝업에서 지급된 캐릭터(조각) 수를 확인

### 기대 결과

10+1 = **총 11회** 소환 결과(캐릭터 11장)가 지급됨

### 실제 결과

- 결과 카드가 **10장만** 표시됨 (보너스 +1회 누락)
- 결과 팝업 타이틀도 "10+1 소환 결과"가 아닌 일반 "소환 결과"로 표기됨
- 획득 합계(예: SR 2 + R 8 = **10개**)로 11개 미달 확인

### 원인 분석

픽업 배너 소환 설정에서 10+1 소환의 반복 횟수(`count`)가 보너스 1회를 포함하지 않은 **10**으로 지정됨.

```javascript
const BANNER = {
  single: { count:1,  cost:200 },
  multi:  { count:10, cost:1800, guaranteeSR:true },  // ← 11이어야 하는데 10
};
// pullBanner(): for(let i=0;i<count;i++){ ... }  → 10회만 반복
```

라벨/가격은 "10+1 · 💎1,800"로 11회 기준이나 실제 지급 루프는 10회만 수행됨.

### 영향 범위

픽업 배너 10+1 소환 이용자 전원. 광고된 11회보다 1회 적게 지급되어 **유료 재화(다이아) 손실 및 확률 기대값 저하**. 픽업 배너의 핵심 상품 신뢰도에 직접 타격.

### 조치 방향

- `BANNER.multi.count`를 11로 수정 (10연 + 보너스 1회)
- `showGachaResult()`의 "10+1 소환 결과" 타이틀 분기 조건(`results.length===11`) 정상 표기 확인
- 1회 / 10+1 각각 지급 수량 경계값 TC 자동화

### 재현 가능 여부

데모에서 직접 재현 가능 — 픽업 배너에서 10+1 소환 후 결과 카드 수를 셈

| `attachments/BUG-004_multi_10cards.png`  | 10+1 소환 결과 — 카드 10장만 지급 (11장이어야 함, 버그) |
