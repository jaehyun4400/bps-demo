# QA 회고 (Retrospective) — BPS 포트폴리오

**프로젝트**: Blades of Plumflower Beneath Snow (BPS)
**대상**: v1.2.0 / v1.3.0 업데이트
**작성자**: 박재현
**최종 수정일**: 2026-06-25

> 본 문서는 BPS 포트폴리오의 **테스트 범위를 정의**하고, 실제 모바일 라이브 게임 QA였다면
> 추가로 수행했을 검증과 다음 개선 계획을 정리한 회고입니다.
> "무엇을 못 했나"의 변명이 아니라 **"무엇을 의도적으로 범위에 넣고/뺐는가"**의 기록입니다.

---

## 1. 프로젝트 성격 및 테스트 범위 정의

본 프로젝트는 **QA 포트폴리오용 플레이어블 프로토타입**으로, 실제 배포 빌드(APK/IPA) 대신
**웹 기반 모바일 시뮬레이션** 형태로 제작했다. 이 선택의 결과로 테스트 범위를 다음과 같이 정의했다.

| 구분 | 범위 | 사유 |
|---|---|---|
| **In Scope** | 튜토리얼성 진입, 재화 지급/차감, 소환·픽업 확률/천장, 전투·보상 루프, 출석·우편·업적·이벤트, 업데이트 회귀, 데이터 영속(저장/로드) | 모바일 라이브 게임에서 유저 신뢰·과금·이탈에 직결되는 핵심 품질 영역 |
| **Out of Scope** | 실기기 발열·메모리·배터리, OS 권한, 스토어 결제 실거래, 푸시/딥링크, 서버 인프라·로그, 크래시·성능 프로파일링 | 실제 빌드·서버·PG 연동이 전제되어야 하는 영역으로, 프로토타입 범위 밖 |

> 이 범위 정의는 핑계가 아니라 **테스트 계획의 일부**다.
> 실제 게임 QA에서도 "무엇을 테스트하지 않을지"를 명시하는 것이 계획서의 핵심이다 ([TEST_PLAN](./1.3/TEST_PLAN.md) §2.2 참조).

## 2. 잘 된 점 — 이 포트폴리오가 증빙하는 QA 역량

1. **의도적 결함(planted bug) 설계 → 재현 → 영향도 분석의 역설계**
   - 코드에 결함을 직접 심고(BUG-001 출석 사이클 리셋 누락, BUG-002 10+1 소환 카운트 오류),
     재현 절차·기대/실제 결과·심각도·**비즈니스 영향**(유료 재화 손실, 확률 기대값 저하, 상품 신뢰도 타격)까지 문서화했다.
   - "버그를 우연히 발견하는 것"을 넘어 **"결함이 어디서 왜 터지는지 구조적으로 아는 것"**을 보여주려는 의도.

2. **자동화 기반 검증(Playwright)**
   - 출석 7일 누적 시뮬레이션, 픽업 천장(49→50회) 도달, 자동 팝업/뱃지 노출을 스크립트로 재현·검증했다.
   - 수동 재현이 오래 걸리는 시간/누적 의존 시나리오를 자동화로 커버.

3. **요구사항 추적성(Traceability)**
   - REQ ↔ TC ↔ BUG를 매핑하여([TRACEABILITY](./1.3/TRACEABILITY.md)) 커버리지 누락과 변경 영향 범위를 추적 가능하게 했다.

4. **업데이트 회귀 관점**
   - v1.2 → v1.3 변경점(시즌 전환, 출석/픽업 신규, 저장 prefix 마이그레이션) 기준으로 회귀 범위를 잡고,
     기존 기능(소환/재화/저장)이 깨지지 않는지 회귀 TC로 검증했다.

## 3. 한계 — 실서비스 QA였다면 추가했을 검증

웹 프로토타입의 구조적 한계상 아래 영역은 범위에서 제외했으나, 실제 라이브 게임 QA였다면 다음을 수행했을 것이다.

| 영역 | 추가했을 검증 |
|---|---|
| **실기기 / 환경** | 저사양·고사양 단말 매트릭스, 화면비(노치/펀치홀) 대응, 발열·메모리·프레임드랍, 백그라운드→포그라운드 복귀, 터치 오입력 |
| **서버 / 데이터** | 서버 응답 지연·실패 시 재화 정합성, 멱등성(중복 요청 방어), 서버-클라 시각 불일치, 로그 기반 지급 검증 |
| **결제** | 스토어 영수증 검증, 결제 중단·환불·중복 결제, 미지급 복구 플로우 |
| **운영 / 푸시** | 푸시 수신·딥링크 진입, 점검 공지·강제 업데이트, 타임존별 일일/주간 초기화 경계 |
| **안정성** | 크래시 수집(crashlytics 가정), ANR, 장시간 구동 메모리 누수 |

> 특히 **재화 중복 지급(멱등성)**과 **서버-클라 동기화**는 수집형 RPG에서 과금 신뢰도와 직결되는
> 최상위 리스크로, 실서비스라면 P1으로 별도 테스트 스위트를 구성했을 것이다.
> (참고: v1.2.0 BUG-001 "재화 교환 연속 터치 시 중복 지급"은 이 리스크의 클라이언트측 단면을 의도적으로 재현한 사례다.)

## 4. 다음 개선 계획

본 포트폴리오를 더 설득력 있게 만들기 위한 다음 작업 우선순위.

1. **결함 다양성 확보** — 현재 v1.3 결함이 Major 2건에 집중. Critical(서비스 중단급)·Minor(표시 오류급)까지
   심각도 스펙트럼을 넓혀, 트리아지(triage)와 우선순위 판단 과정을 더 풍부하게 보여준다.
2. **자연 발생 결함 발견 사례 추가** — 의도적으로 심은 버그 외에, 구현 과정에서 **실제로 발견·분석한 결함** 사례를
   1건 이상 기록하여 "탐지력"을 직접 증명한다.
3. **재현 영상/GIF 증빙** — 정적 스크린샷 위주의 증빙을 보강해, 핵심 버그는 짧은 재현 영상으로 제공한다.
4. **대표 결함 카드화** — 면접관이 30초 안에 읽을 수 있도록 핵심 버그를
   "증상 / 재현 / 유저 영향 / 비즈니스 영향 / 우선순위 근거" 1장 카드로 요약한다.
