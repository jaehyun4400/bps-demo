### GitHub 배포

개발 파일을 친구에게 공유했더니 이미지가 엑스박스(깨진 이미지)로 보이는 문제가 있었다. HTML 파일만 단독으로 전달하면 `portraits_unified/`, `sd_chars/` 같은 상대경로 폴더가 없으니 당연한 일이었다.

해결책으로 GitHub Pages를 이용한 무료 정적 호스팅을 선택했다. 회사 GitHub 계정에 개인 레포를 만드는 것에 대한 우려가 있었는데, GitHub 조직(org) 구조상 개인 레포와 회사 org 레포는 완전히 분리되어 있어 문제없음을 확인했다. 또한 회사 컴퓨터이기 때문에 git config를 `--global`이 아닌 `--local`로 설정해 bps 폴더에만 적용되도록 했다.

배포 시 `.gitignore`에서 루트 경로 지정 없이 파일명만 기재했더니, `portraits_unified/` 폴더 안의 동명 파일까지 제외되는 문제가 발생했다. `/gungny.png` 형식으로 루트 경로를 명시해 해결했다.

### 자동돌파(autoDive) 버그 수정

D2-10(던전2 보스), D3-10(던전3 보스) 클리어 후 자동돌파가 멈추는 버그가 발생했다.

**원인 분석:**

`showResult()` 함수 내 `isLast` 계산 로직이 문제였다.

```javascript
// 버그 코드
const isLast = S.stage === (DUNGEONS[S.dungeon].stageCount || 10) - 1;
```

D1~D4는 `stageCount`가 없어 기본값 10이 적용된다. 따라서 stage === 9(보스 스테이지)이면 `isLast = true`가 되어버렸고, 자동돌파 블록인 `if(win && S.autoDive && !isLast)` 조건이 항상 false가 되어 실행되지 않았다.

**수정:**

```javascript
// 수정 코드
const isLastStage = S.stage === (DUNGEONS[S.dungeon].stageCount || 10) - 1;
const isLast = isLastStage && S.dungeon === DUNGEONS.length - 1;
```

마지막 스테이지이면서 **동시에 마지막 던전**일 때만 `isLast = true`가 되도록 수정했다. D7 마지막 스테이지에서만 로비로 복귀하게 되었다.

### 보호막 시스템 기획 논의 및 구현

혜 귀인의 패시브 "HP 60% 이하 시 보호막 발동"에 대한 상세 설계를 논의하고 전부 구현 완료했다.

**확정 스펙:**

- **보호막 수치:** 캐릭터 최대 체력의 15% 고정
- **데미지 흡수 방식:** 쉴드가 HP보다 먼저 차감 (버퍼 역할)
  - 예) HP 80%, 쉴드 15% 활성 → 70% 데미지 수신 → 쉴드 15% 선차감 → HP에 55% 적용 → 최종 HP 25%
- **재발동 쿨타임:** 20초
- **쿨타임 경과 시:** HP가 60% 이하이면 즉시 자동 재발동
- **쉴드 캡:** 쿨타임 경과 시 잔여 쉴드 수치와 무관하게 15%로 리셋 (누적 불가)
- **스테이지 진입 시:** 쿨타임 초기화, HP 60% 이하면 즉시 발동
- **HP 강화와 연동:** 의도된 설계 — HP를 올릴수록 쉴드량도 함께 증가, 유저에게 "강공 vs 버티기" 선택권 부여
- **쿨타임 중 HP 변동:** HP가 60% 이상으로 회복됐다가 다시 60% 이하가 되어도 쿨타임에는 영향 없음

### HP 오브 UI 전면 교체 (도넛 → 디아블로 스타일)

기존 `conic-gradient` 기반 도넛 차트 방식을 버리고 Canvas 기반 액체 구체로 교체했다. 디아블로 시리즈의 HP 오브에서 영감을 받은 UI다.

**구현 방식:**

- `<canvas>` 72×72px 요소에 `requestAnimationFrame` 루프로 매 프레임 드로잉
- 파도치는 액체 상단은 `Math.sin()` 으로 표현 — x 위치마다 위상을 다르게 줘서 물결 느낌 구현
- HP 비율에 따른 색상 전환: `#ef4444`(빨강, 50%↑) → `#f97316`(주황, 25%↑) → `#dc2626`(진빨강)
- 유리 광택: `createRadialGradient`로 상단 좌측에 흰색 하이라이트
- 내부 림: 가장자리 어두운 그라디언트로 입체감 표현

**보호막 시각화:**

- 쉴드 활성 시 HP 액체 위에 하늘색(`rgba(160,220,255,0.8)`) 액체 레이어 추가
- HP와 쉴드가 오브 내부에서 아래부터 쌓이는 구조 (HP 빨강 + 쉴드 파랑)
- 외곽 `.hp-orb-shield-ring` CSS 요소:
  - 활성: 파란 펄스 애니메이션 (`shieldRingPulse`)
  - 쿨타임: 회색 dim 링 (`.cooldown` 클래스)
  - 소진: 확산 파열 이펙트 (`shieldShatter` keyframe → scale 1.45까지 확대 후 opacity 0)

### 버그 수정 — 무기 미착용 시 SD 캐릭터 오표시

무기를 착용하지 않은 상태인데 `sword_girl.png`가 노출되는 문제를 발견했다. 원인은 `getWpType()` 함수의 폴백값이 `'sword'`로 하드코딩되어 있었기 때문이다.

```js
// 수정 전
function getWpType() { const w=getWeapon(); return w?w.type:'sword'; }
// 수정 후
function getWpType() { const w=getWeapon(); return w?w.type:''; }
```

빈 문자열을 반환하면 `getSDCharImg()`의 기본 분기로 떨어져 `normal_girl.png`가 정상 표시된다. 추가로 `getWpType()==='sword'` 비교를 사용하는 `doAttackAnim()`도 영향 없음을 확인했다 — 무기 없을 때 검기 애니메이션이 나오지 않는 것이 오히려 올바른 동작이다.

### SD 치비 캐릭터 시스템

필드 전투 화면에서 기존 초상화(portrait) 대신 SD 치비 캐릭터 이미지를 사용하도록 개선했다. 무기 종류에 따라 이미지가 자동으로 전환되고, 적의 위치에 따라 좌우 반전도 자동으로 처리된다.

**SD 이미지 매핑**

| 무기 타입 | SD 이미지       | 필드 크기 | 보스 크기 |
| --------- | --------------- | --------- | --------- |
| sword     | sword_girl.png  | 45px      | 50px      |
| bow       | bow_girl.png    | 45px      | 50px      |
| orb       | mage_girl.png   | 45px      | 50px      |
| (미장착)  | normal_girl.png | 45px      | 50px      |

이미지 배경 제거 작업이 예상보다 까다로웠다. Python Pillow의 flood-fill 방식을 사용했는데, tolerance를 높이면 캐릭터 팔/손 영역까지 제거되고, 낮추면 배경이 남는 문제를 반복해서 조율해야 했다. 최종적으로 `tolerance=55` + HLS 엣지 정리(L>0.82, S<0.35, 투명 이웃 필요) + 소형 컴포넌트 제거(size<500 or avg_L>0.76) 조합으로 해결했다.

구현 중 두 가지 추가 버그가 발생했다:

- **좌우 반전 시 흰색 깜빡임:** CSS `transition:transform`과 피격 시 `brightness` 효과가 충돌. transition 제거 및 brightness 오버라이드 삭제로 해결.
- **캐릭터 변경 시 오라 미반영:** `updateSDChar()`가 이미지 경로만 갱신하고 오라 클래스(SSR/SR/R 등급별 glow 효과)는 갱신하지 않던 문제. 오라 클래스와 이름 텍스트를 함께 갱신하도록 확장해 해결.

보스 전투에도 SD 캐릭터를 적용했다. 캐릭터/무기 변경 핸들러에서 `isFieldMode()` 조건을 제거해 보스전 중에도 즉시 반영되도록 했다.

### 근거리 무기 vs 원거리 몬스터 버그 수정

근거리 무기 착용 시 원거리 몬스터를 공격할 수 없는 버그가 있었다.

**원인:** 원거리 몬스터는 플레이어로부터 130px(RANGED_STOP) 거리에서 멈추는데, 근거리 공격 범위는 80px(MELEE_RANGE)이었다. 플레이어가 80px 안으로 들어가지 않으면 공격 자체가 불가능한 구조였다.

**해결:** `S.playerTargetIdx` 상태 필드를 추가해 락온 타겟을 관리하도록 했다. `pickPlayerTarget()` 함수를 구현해 현재 타겟이 살아있으면 유지하고, 사망 시 가장 가까운 적으로 전환한다. `fieldUpdate()` 루프에서 조이스틱 미조작 + 근거리 무기 + 타겟이 MELEE_RANGE 밖인 경우 플레이어가 자동으로 타겟을 향해 이동하도록 추가했다.
